Airo OPS Deploy Handoff
=======================

Project:
- Repo: Airo/AirOS-New
- Main branch: main
- GitHub remote exists and is already configured
- "OPS" means the Origin Point Server, the Raspberry Pi

What was added:
- scripts/deploy-pi.mjs
- package.json script: "deploy:pi"

What deploy:pi now does:
- Reads .env.local and environment variables
- Runs git fetch --all --prune and git pull --ff-only
- Installs dependencies with npm ci when package-lock.json exists, otherwise npm install
- Ensures two systemd services exist on OPS:
  - preview service for the HTTPS Airo server
  - tunnel service for the Cloudflare tunnel
- Restarts both services
- Waits for a valid trycloudflare URL to change
- Updates the GitHub Gist only when the valid public URL changed
- When OPS has a desktop session, shows a simple status window with steps like:
  - Installing AirOS Update...
  - Restarting AirOS Server...
  - Running Cloudflare Tunnel...
  - Updating Gist...

Default OPS service assumptions:
- preview service name: env AIRO_PREVIEW_SYSTEMD_SERVICE, or AIRO_SYSTEMD_SERVICE, or airo
- tunnel service name: env AIRO_TUNNEL_SYSTEMD_SERVICE, or "<preview>-tunnel"
- service user: env AIRO_SERVICE_USER, or $USER, or pi
- working directory: env AIRO_SERVICE_WORKDIR, or the current repo root
- preview start command: env AIRO_PREVIEW_EXEC_START, or AIRO_SERVICE_EXEC_START, or /usr/bin/npm run preview:https
- tunnel start command: env AIRO_TUNNEL_EXEC_START, or /usr/bin/node <repo>/scripts/start-public-tunnel.mjs
- cloudflared binary: env CLOUDFLARED_BIN, or AIRO_CLOUDFLARED_BIN, or /usr/bin/cloudflared

Important runtime files already used by the repo:
- .current-public-url
- .current-server-url
- .current-gist-id
- .current-latest-endpoint

Cloudflare / Gist notes:
- scripts/start-public-tunnel.mjs writes the latest public URL
- scripts/start-public-tunnel.mjs now supports CLOUDFLARED_BIN / AIRO_CLOUDFLARED_BIN
- scripts/update-gist-latest-url.mjs updates the Gist using GITHUB_TOKEN or GH_TOKEN
- deploy:pi only updates the Gist when the new URL is a valid https://<name>.trycloudflare.com hostname

Important fix versus the old handoff:
- deploy:pi no longer assumes the tunnel will change on its own
- deploy:pi now manages the OPS preview service and tunnel service explicitly
- the old "service exists" check was replaced with a real unit-file existence check
- deploy:pi no longer treats any random https URL as a valid Cloudflare result
- deploy:pi can now show a simple OPS progress window when DISPLAY or WAYLAND_DISPLAY is available

OPS setup note:
- On a fresh OPS box, deploy:pi will create /etc/systemd/system/<preview>.service if missing
- It will also create /etc/systemd/system/<tunnel>.service if missing
- The service creation step uses sudo
- If sudo prompts for a password, that is expected on first run

Suggested update flow:
1. Edit on Mac
2. Commit and push to GitHub
3. On OPS, run: npm run deploy:pi

What not to change casually:
- The tracked URL files
- The Gist update logic unless the URL storage format changes
- The preview/tunnel service naming unless you also update the related env vars
- CLOUDFLARED_BIN handling unless you verify both OPS and Mac flows

Known OPS caveats:
- git pull --ff-only will fail on a dirty OPS checkout
- npm ci will fail if package-lock.json and package.json drift
- If cloudflared is not installed at /usr/bin/cloudflared, set CLOUDFLARED_BIN explicitly
- If OPS should use a different start command, set AIRO_PREVIEW_EXEC_START and/or AIRO_TUNNEL_EXEC_START
- If OPS is running headless with no desktop session, the status window will not appear, but deploy still works

If another agent edits this area:
- Keep deploy-pi aligned with scripts/start-public-tunnel.mjs
- Do not revert to a single-service assumption unless the tunnel becomes embedded in the main service
- If changing service startup commands, verify:
  - preview service starts
  - tunnel service starts
  - .current-public-url updates
  - the Gist is updated only after a valid new public URL appears
